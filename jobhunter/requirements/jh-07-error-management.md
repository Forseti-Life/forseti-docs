# JobHunter Requirements — JH-07: Error Queue & Admin Management

**Status summary:** 6 of 6 requirements COMPLETED, 0 PARTIAL, 0 GAP  
**Last updated:** 2026-02-27  
**Module:** `job_hunter` (forseti.life)

## Overview

This subsystem provides centralized error capture, storage, and administrative review for all processing errors generated by JobHunter's queue workers and services. It exposes a dedicated admin interface for viewing, filtering, and resolving errors, and an API endpoint that surfaces the current error count for use in dashboard badges and monitoring.

## Requirements

### REQ-07.1 — Central error capture

**Status:** COMPLETED  
**Code path:** `src/Service/ErrorQueueService.php`

- The system must capture all processing errors from queue workers and key services to a central error queue via `ErrorQueueService`.
- Every captured error must store: timestamp, error type/code, context (job posting ID, user ID, queue item ID where applicable), error message, and full stack trace.
- `ErrorQueueService` must be injectable and used consistently across all queue workers via `QueueWorkerBaseTrait`.

---

### REQ-07.2 — Admin error list view

**Status:** COMPLETED  
**Code path:** `src/Controller/ErrorQueueController.php`, route `job_hunter.error_queue_admin`

- An administrator must be able to view all captured errors at `/admin/job-hunter/errors`.
- The list must display: timestamp, error type, affected entity (job/user), and resolution status.
- The list must support filtering by: resolution status (open / fixed), error type, and date range.
- Access must require the `administer job hunter` permission.

---

### REQ-07.3 — Admin error detail view

**Status:** COMPLETED  
**Code path:** `src/Controller/ErrorQueueController.php`, route `job_hunter.error_queue_detail`

- An administrator must be able to view full detail for a single error at `/admin/job-hunter/errors/{error_id}`.
- The detail view must display: all stored error fields (see REQ-07.1), the queue item payload at the time of failure, and the resolution history.

---

### REQ-07.4 — Mark error as fixed

**Status:** COMPLETED  
**Code path:** `src/Controller/ErrorQueueController.php`

- An administrator must be able to mark an error as "fixed" from the detail view or the error list.
- Marking an error as fixed must record the timestamp and the acting administrator's user ID.
- Fixed errors must remain visible in the error list with a "fixed" status indicator (not deleted).

---

### REQ-07.5 — Error count API endpoint

**Status:** COMPLETED  
**Code path:** `src/Controller/ErrorQueueController.php`, route `job_hunter.error_count_api`

- The system must expose the current count of unresolved errors via a JSON API endpoint at `/api/job-hunter/error-count`.
- The endpoint must return the count in a machine-readable JSON structure.
- This endpoint must be usable by front-end components (navigation badges, dashboards) via AJAX without a page reload.
- Access to the endpoint must be limited to authenticated users with appropriate permissions.

---

### REQ-07.6 — Required error fields

**Status:** COMPLETED  
**Code path:** `src/Service/ErrorQueueService.php`

- Every error record must include all of the following fields:
  - `timestamp` — ISO 8601 datetime of when the error occurred
  - `error_type` — machine-readable type identifier (e.g., `bedrock_api_failure`, `pdf_render_error`)
  - `context` — structured context array containing at minimum: user ID, queue worker name, and affected entity ID/type
  - `message` — human-readable error message
  - `stack_trace` — full PHP stack trace at the time of failure
  - `status` — resolution status: `open` or `fixed`
- All fields must be present; partial error records must not be accepted.

---

## Known Gaps

No functional gaps identified in this subsystem. The following quality improvements are recommended:

- Error records are never automatically purged; a retention policy (e.g., auto-delete fixed errors older than 90 days) should be implemented.
- No alerting mechanism exists (e.g., email notification when error count exceeds a threshold).
- The error list does not support pagination for large error volumes.

## Test Coverage

No unit tests exist for `ErrorQueueService` or `ErrorQueueController`. Estimated coverage: **0%** for this subsystem. Priority test targets: `ErrorQueueService::captureError()` field validation, the error count API endpoint response format, and the mark-as-fixed state transition.
