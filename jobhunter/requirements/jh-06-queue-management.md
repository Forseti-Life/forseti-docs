# JobHunter Requirements — JH-06: Queue Management & Worker Infrastructure

**Status summary:** 8 of 8 requirements COMPLETED, 0 PARTIAL, 0 GAP  
**Last updated:** 2026-02-27  
**Module:** `job_hunter` (forseti.life)

## Overview

This subsystem provides the asynchronous processing backbone for all AI-intensive operations in JobHunter. It defines five queue workers, a shared `QueueWorkerBaseTrait` for retry/logging behavior, and an administrator UI at `/jobhunter/queue-management` for monitoring, pausing, retrying, and deleting queue items.

## Requirements

### REQ-06.1 — Asynchronous processing via queue workers

**Status:** COMPLETED  
**Code path:** `src/Plugin/QueueWorker/`

- All AI-intensive and long-running operations must be processed asynchronously via Drupal Queue API workers.
- The following queue workers must exist and be registered:
  - `ResumeTextExtractionWorker` — extracts text from uploaded resume files
  - `ResumeGenAiParsingWorker` — parses extracted text into structured JSON via AWS Bedrock
  - `ResumeTailoringWorker` — generates AI-tailored resume content for a specific job
  - `CoverLetterTailoringWorker` — generates AI-tailored cover letter for a specific job
  - `JobPostingParsingWorker` — parses raw job posting data into structured fields
  - `ApplicationSubmitterQueueWorker` — dispatches application submission tasks (partial; see JH-04)
- Workers must be discoverable as Drupal plugin annotations under `@QueueWorker`.

---

### REQ-06.2 — Automatic retry with 3-attempt limit

**Status:** COMPLETED  
**Code path:** `src/Traits/QueueWorkerBaseTrait.php`

- Every queue worker must automatically retry a failed item up to 3 times before marking it as suspended.
- Each retry attempt must be logged with: attempt number, worker name, item identifier, and failure reason.
- The retry count must be stored on the queue item itself so it survives process restarts.
- `QueueWorkerBaseTrait` must provide the retry counting, logging, and suspension logic for all workers via a shared trait.

---

### REQ-06.3 — Queue management admin UI

**Status:** COMPLETED  
**Code path:** `src/Controller/WorkflowController.php`, route `job_hunter.queue_management`

- An administrator UI must be available at `/jobhunter/queue-management`.
- The UI must display all queue workers with their current counts: pending items, items in progress, and suspended (failed) items.
- Access to this UI must require the `administer job hunter` permission.

---

### REQ-06.4 — Manual retry of suspended items

**Status:** COMPLETED  
**Code path:** `src/Controller/WorkflowController.php`

- From the queue management UI, an administrator must be able to manually retry a suspended queue item.
- Retrying a suspended item must reset its retry count and re-enqueue it.
- The UI must display the failure history for each suspended item before re-queuing.

---

### REQ-06.5 — Manual suspension and deletion of queue items

**Status:** COMPLETED  
**Code path:** `src/Controller/WorkflowController.php`

- From the queue management UI, an administrator must be able to manually suspend an active queue item (prevent it from being picked up by the next cron run).
- An administrator must be able to delete individual queue items permanently.
- An administrator must be able to delete associated files generated by a queue item (e.g., intermediate resume files).

---

### REQ-06.6 — Pause and resume all queue processing

**Status:** COMPLETED  
**Code path:** `src/Controller/WorkflowController.php`

- An administrator must be able to pause all JobHunter queue processing system-wide with a single action.
- While paused, no queue worker must claim or process new items during cron runs.
- An administrator must be able to resume processing with a single action.
- The pause/resume state must persist across page loads and cron invocations.

---

### REQ-06.7 — GenAI cache clearing

**Status:** COMPLETED  
**Code path:** `src/Controller/WorkflowController.php`

- An administrator must be able to clear the GenAI response cache from the queue management UI.
- Clearing the cache must force all subsequent AI operations to make fresh API calls to AWS Bedrock rather than returning cached responses.
- The cache-clear action must be logged.

---

### REQ-06.8 — Queue status AJAX polling

**Status:** COMPLETED  
**Code path:** `src/Controller/UserProfileController.php`, `src/Controller/WorkflowController.php`

- The queue management UI must expose queue status (pending/processing/failed counts per worker) via an AJAX endpoint suitable for dashboard polling.
- The endpoint must return JSON and be callable without a full page reload.
- Status counts must refresh automatically in the UI without requiring a manual refresh.

### REQ-06.9 — Per-item queue failure isolation

**Status:** GAP
**Code path:** `src/Plugin/QueueWorker/ResumeTailoringWorker.php`

- A failure processing a single queue item must not suspend processing for all other users' items in the same queue.
- When a queue worker encounters an unrecoverable item (e.g., malformed JSON payload, AI service error after max retries), it must move the item to the dead-letter / error queue and continue processing the next item.
- `SuspendQueueException` must NOT be thrown for single-item failures; it is reserved for infrastructure-level failures that affect all items (e.g., database unavailable, queue service unreachable).
- Dead-lettered items must be visible in the existing error queue admin UI (REQ-07.x).

**Acceptance criteria:**
1. Injecting a malformed JSON payload as a tailoring queue item causes only that item to move to the error queue; other users' tailoring items continue processing normally.
2. `SuspendQueueException` is never thrown in `ResumeTailoringWorker` for payload-level failures.
3. QA verifies by submitting a bad payload and confirming other items complete within the normal SLA.
